[PATCH v3 00/42] KVM: arm64: Revamp Fine Grained Trap handling

43 messages 1 participants Started: 2025-04-26
本邮件线程讨论了关于 KVM（内核虚拟机）在 arm64 架构下的细粒度陷阱（Fine Grained Trap，FGT）处理的改进，特别是引入了 FEAT_FGT2 特性。

1. **原始 Patch/问题内容**：
本次讨论的核心是对 KVM 的细粒度陷阱处理进行重构，特别是对 FEAT_FGT2 的支持。该系列补丁旨在改进 KVM 对细粒度陷阱的处理方式，以便更好地支持新的 ARM 架构特性。

2. **之前讨论要点**：
在之前的讨论中，Marc Zyngier 提到该补丁系列的规模显著增加，主要是因为添加了对 FEAT_FGT2 的全面支持。补丁还包括对现有 FGT 描述和处理原语的重构，以减少冗余代码。开发者们讨论了如何确保在 KVM 中正确处理新引入的寄存器和指令。

3. **本周的新讨论、进展或结论**：
本周的讨论集中在多个补丁的具体实现上，包括：
- 添加了对 FEAT_FGT2 寄存器的支持和描述。
- 引入了新的 FGT 处理机制，以便在 KVM 中进行上下文切换。
- 讨论了如何使用计算的掩码来设置 FGT 寄存器，并确保它们与 KVM 的特性映射一致。
- 还增加了对 PSB CSYNC 和 TSB CSYNC 指令的处理，确保这些指令能够正确触发陷阱。

总的来说，这一系列补丁的目标是提高 KVM 对 ARM 架构新特性的支持，确保虚拟机在处理细粒度陷阱时的稳定性和性能。

在本周的新讨论中，参与者对补丁进行了进一步审查和修改建议。Markus Elfring 提出了对 Coccinelle 脚本的改进建议，Luo Jie 表示将根据反馈更新补丁。Yury Norov 和 Russell King 继续讨论 `FIELD_MODIFY()` 与 `_replace_bits()` 的区别，强调了编译器在捕捉错误方面的优势。整体来看，`FIELD_MODIFY()` 得到了越来越多的认可，预计将在 Qualcomm 的以太网驱动中使用。

[PATCH v3 00/17] KVM: arm64: Recursive NV support

18 messages 1 participants Started: 2025-04-23
本邮件线程讨论了针对 KVM 的 ARM64 架构中递归虚拟化（NV）支持的补丁集（PATCH v3 00/17）。该补丁的核心是实现对 VNCR_EL2 的虚拟化支持，涉及到如何在嵌套虚拟化环境中管理 TLB（Translation Lookaside Buffer）和处理相关的内存映射。

**原始补丁/问题内容**：
补丁主要集中在如何处理 VNCR_EL2 的虚拟化，特别是如何在 L1 和 L2 之间正确映射 VNCR 页，确保在访问系统寄存器时能够正确处理上下文切换。

**之前讨论要点**：
在历史讨论中，Marc Zyngier 提到，VNCR 页的管理涉及到 TLB 的管理和内存映射的有效性，特别是在 L1 和 L2 之间的交互中，必须考虑到多种可能导致 VNCR 页不可访问的情况，如内存被换出或权限变化。

**本周的新讨论、进展或结论**：
本周的讨论中，Marc Zyngier 提出了多个补丁，逐步实现了 VNCR_EL2 的支持，包括：
1. 为 VNCR_EL2 添加布局和处理逻辑。
2. 处理 VNCR_EL2 触发的故障，确保在缺失映射时能够正确处理。
3. 实现 TLBI（Translation Lookaside Buffer Invalidation）指令的支持，以便在需要时能够有效地清除 TLB。
4. 允许用户空间请求 KVM_ARM_VCPU_EL2*，并更新文档以反映新的功能。

整体来看，这些补丁使得 NV 支持逐步完善，Marc 表示尽管尚未达到完全优化的状态，但当前的实现已经能够支持 L4 客户机的运行。

[PATCH v3 1/1] KVM: arm64: Allow cacheable stage 2 mapping using VMA flags

16 messages 5 participants Started: 2025-04-07
本邮件线程讨论了一个针对 KVM（内核虚拟机）在 arm64 架构上允许使用 VMA（虚拟内存区域）标志进行可缓存的二级映射的补丁（PATCH v3 1/1）。该补丁旨在改善 KVM 对物理页框号（PFNMAP）内存的处理，特别是在主机的一级映射中使用可缓存属性时。

在历史讨论中，参与者们探讨了 KVM 如何处理 PTE（页表项）复制时的情况，尤其是当物理地址没有对应的结构页时，KVM 需要假设最坏情况，即无法进行缓存刷新。此外，讨论还涉及了是否需要向用户空间暴露 KVM 是否支持可缓存 PFNMAP 的能力。

本周的新讨论中，Ankit Agrawal 提醒大家关注总结和后续步骤，并询问 Marc 是否仍然认为应该添加“可缓存 PFNMAP” KVM 能力。Oliver Upton 强调，用户空间需要了解内存的缓存一致性，以便在进行虚拟机迁移时做出正确的决策。Jason Gunthorpe 和其他参与者对是否需要添加 memslot 标志进行了激烈讨论，认为这可能会增加复杂性，而 Sean Christopherson 则反对添加 memslot 标志，认为仅依靠 KVM 能力就足够保护 VMM。

总体来看，讨论围绕如何在 KVM 中合理管理可缓存内存映射展开，参与者们对补丁的必要性和实现细节持有不同看法，尚未达成一致。
