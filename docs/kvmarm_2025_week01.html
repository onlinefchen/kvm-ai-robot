[PATCH v1 00/13] KVM: Introduce KVM Userfault

2 messages 2 participants Started: 2024-12-24
本邮件讨论的主题是关于 KVM Userfault 的引入，涉及到性能优化和内存管理的相关问题。

1. **原始 patch/问题的内容**：该补丁（[PATCH v1 00/13]）旨在引入 KVM Userfault，作为一种性能优化特性，主要用于处理虚拟机内存访问的故障。补丁的目标是提升 KVM 在处理用户故障时的效率，尤其是在不使用 gmem 的情况下。

2. **之前的讨论要点**：在历史讨论中，参与者对该特性的目的进行了澄清，认为它主要是性能优化，而非替代现有的 userfaultfd。讨论中提到，KVM Userfault 不能完全替代 userfaultfd，尤其是在涉及到远程故障的情况下。参与者强调了在不同架构和组件中对内存访问的处理复杂性。

3. **本周的新讨论、进展或结论**：在本周的讨论中，James Houghton 和 Peter Xu 进一步探讨了 KVM Userfault 的实现细节，确认其在 gmem 和非 gmem 情况下的不同应用。Houghton 表示，KVM Userfault 将不会取代 userfaultfd，因为这会增加复杂性，并且在某些情况下，用户空间的映射和支持仍然是必要的。他们还讨论了将 KVM Userfault 集成到 QEMU 的潜在好处，并提到在 GCE 环境中已经观察到了一些性能提升。整体来看，讨论集中在如何优化和实现这一新特性，以便更好地支持虚拟化环境中的内存管理。

[RFC PATCH v2 55/58] drivers/iommu: Add deferred map_sg operations

5 messages 3 participants Started: 2024-12-19
本邮件线程讨论了一个关于 IOMMU 驱动的补丁，标题为「[RFC PATCH v2 55/58] drivers/iommu: Add deferred map_sg operations」。该补丁旨在引入延迟的 `map_sg` 操作，以提高 IOMMU 映射的效率。

在历史讨论中，Mostafa Saleh 提出了这个补丁的初衷，Robin Murphy 则质疑是否有必要引入复杂的机制，建议可以通过使 `->map_pages` 异步化来简化操作，并将昂贵的部分整合到 `->iotlb_sync_map` 中。然而，Mostafa 认为现有的 `iotlb_sync_map` 缺乏足够的上下文，可能需要创建每个域的延迟映射列表，这会影响并发性。

在本周的新讨论中，Jason Gunthorpe 对补丁的具体实现进行了深入分析，指出当前的 IOMMU API 在执行 `iommu_map_sg()` 时效率低下，建议引入新的 `add_deferred_map_sg()` 操作来优化映射过程。尽管这种方法增加了三个新的 IOMMU 操作，但在当前情况下只被该驱动使用。Jason 还提到，未来应朝着不需要 `sg` 的快速 DMA 映射路径发展，建议将超调用批处理并执行，以提高效率。

总体来看，本周的讨论进一步明确了补丁的实现细节和潜在的优化方向，参与者们对如何平衡复杂性与性能进行了深入探讨。

[RESEND RFC PATCH v1 2/5] arm64: Add BBM Level 2 cpu feature

6 messages 4 participants Started: 2024-12-19
本邮件讨论的主题是关于在 arm64 架构中添加 BBM Level 2 CPU 特性的一项补丁（patch）。该补丁旨在处理与 TLB（Translation Lookaside Buffer）相关的冲突中止（conflict abort）问题，特别是在 KVM（Kernel-based Virtual Machine）环境中。

在历史讨论中，Ryan Roberts 提出了对处理冲突中止的质疑，认为在 KVM 之外处理这些情况是多余且复杂的，建议将补丁的范围限制在 KVM 支持上。Will Deacon 和其他参与者对此表示关注，认为需要对 BBM Level 2 的实现进行更深入的讨论。

在本周的新讨论中，Jonathan Cameron 提到华为的实现可能会在改变块大小时报告 TLB 冲突中止，这符合架构规范。Marc Zyngier 认为处理 TLB 冲突中止非常复杂，建议只关注那些不会产生此类中止的实现。Will Deacon 赞同这一观点，并表示不会放弃对不符合期望的 CPU 的支持，但也不打算为其维护新的优化。

Ryan Roberts 最后指出，如果支持所有符合架构规范的 BBM Level 2 实现，可能需要放弃补丁系列中的最后一部分，但目前的方向是仅对不会引发冲突中止的实现进行优化。他将与 Miko 合作进一步完善补丁并重新发布。整体来看，讨论集中在如何平衡性能与维护性之间的关系，以及对不同实现的支持策略。
