[PATCH v5 0/4] KVM: arm64: Errata management for VM Live migration

5 messages 1 participants Started: 2025-01-24
本邮件讨论的主题是关于 KVM（Kernel-based Virtual Machine）在 ARM64 平台上进行虚拟机实时迁移时的错误管理（Errata management）。Shameer Kolothum 提出了一个包含四个补丁的系列更新（PATCH v5 0/4），旨在改进虚拟机迁移过程中对 CPU 错误的管理。

在历史讨论中，Shameer 提到 ARM64 平台的许多错误处理措施依赖于 CPU 的 MIDR/REVIDR 值，迁移到具有不同 MIDR/REVIDR 值的平台时，可能会导致问题。因此，补丁的目标是通过引入新的 hypercall 来获取目标 CPU 的实现信息，从而使来宾内核能够适应不同的 CPU 错误处理。

本周的新讨论中，Shameer 更新了补丁，主要进展包括：
1. **补丁更新**：对补丁进行了多个版本的迭代，解决了之前来自 Marc 和 Oliver 的反馈，增加了获取目标 CPU 实现版本和数量的 hypercall。
2. **新功能**：补丁中引入了新的 hypercall 支持，允许来宾内核根据目标 CPU 实现来启用相关的错误处理措施。
3. **代码实现**：补丁的具体实现涉及多个文件的修改，增加了对目标 CPU 实现的支持，并更新了 KVM 的 hypercall 列表。

整体来看，这一系列补丁旨在增强 KVM 在 ARM64 平台上的错误管理能力，确保虚拟机在迁移过程中能够正确处理 CPU 相关的错误。

[PATCH] KVM: arm64: Flush hyp bss section after initialization of variables in bss

3 messages 2 participants Started: 2025-01-20
本邮件讨论的主题是关于 KVM（Kernel-based Virtual Machine）在 arm64 架构下的一个补丁，旨在初始化 BSS 段后刷新 hypervisor 的 BSS 区域。补丁的主要内容是，在初始化过程中，NVHE（Non-virtual Hypervisor Extension）超管需要使用主机 CPU 特征寄存器的清理值，而这些值存储在 BSS 段中。为了确保 EL2 能够正确读取这些值，补丁建议在初始化完成后一次性刷新整个 BSS 段，而不是逐个刷新变量，以提高效率。

在之前的讨论中，补丁的提出者 Lokesh Vutla 指出，未能刷新 BSS 段会导致 EL2 读取到错误的值，从而引发内核崩溃。Marc Zyngier 对此提出质疑，认为在超管初始化后执行刷新可能会导致访问违规，并建议在 kvm_hyp_init_symbols() 函数中进行刷新操作。

本周的新进展中，Lokesh 认可了 Marc 的意见，决定在补丁的下一版本中将刷新操作移至 kvm_hyp_init_symbols() 函数，并表示之前的测试结果表明逐个刷新变量的方式能正常工作，但最终选择优化为刷新整个 BSS 段。Lokesh 将在后续版本中修正这些问题并重新提交补丁。
