[PATCH v3 1/1] KVM: arm64: Allow cacheable stage 2 mapping using VMA flags

18 messages 6 participants Started: 2025-03-17
本邮件讨论的主题是关于 KVM（内核虚拟机）在 arm64 架构下允许使用 VMA（虚拟内存区域）标志进行可缓存的二级映射的补丁（PATCH v3 1/1）。该补丁旨在解决在没有 FWB（Firmware Write Back）支持的情况下，如何安全地处理 PFNMAP（页面框架号映射）作为非设备内存的问题。

在历史讨论中，参与者们探讨了补丁的实现细节，包括如何在内存插槽注册时要求可缓存的 PFNMAP，以及如何确保虚拟机监控器（VMM）和用户空间之间的映射一致性。讨论中提到，当前补丁依赖于阶段 1 的 VMA 属性，但存在潜在的安全隐患，尤其是在不同主机之间迁移虚拟机时。

在本周的新讨论中，Ankit Agrawal 提出是否可以创建一个失败补丁以在内存插槽创建时进行检查。Sean Christopherson 强调，内存插槽创建时的检查是可选的，真正的检查应在映射时进行。他认为，若没有 FWB，缓存的 PFNMAP 内存应被禁止。Marc Zyngier 则反对使用内存插槽标志，认为缓存性应由内核子系统控制，并建议通过能力枚举来处理缓存映射的支持。

总体而言，讨论围绕如何在不同的硬件条件下安全地处理缓存内存映射展开，参与者们对补丁的实现方式和潜在的安全性问题进行了深入的探讨。

[PATCH 6.6 1/8] KVM: arm64: Calculate cptr_el2 traps on activating traps

16 messages 2 participants Started: 2025-03-21
本邮件线程讨论了与 KVM（Kernel-based Virtual Machine）相关的多个补丁，主要集中在 arm64 架构的改进上。

1. **原始补丁内容**：第一个补丁是关于在激活陷阱时计算 `cptr_el2` 的值，目的是消除在每个 vCPU 结构中存储 `cptr_el2` 的需求，并确保某些陷阱（如来宾是否拥有 FP 寄存器）在每次 vCPU 运行时都能正确设置。

2. **历史讨论要点**：在历史讨论中，补丁的背景信息被详细阐述，包括它如何解决之前版本中的问题，例如在 VHE（Virtualization Host Extensions）模式下的状态管理和陷阱处理。这些补丁旨在提高 KVM 的稳定性和性能，尤其是在处理 SVE（Scalable Vector Extension）和 FPSIMD（Floating Point and SIMD）状态时。

3. **本周的新讨论与进展**：本周的讨论中，多个补丁被添加到 6.6 稳定树中，包括：
- `KVM: arm64: Eagerly switch ZCR_EL{1,2}`：确保在来宾与主机之间切换时，ZCR_EL1 和 ZCR_EL2 的值能够及时更新。
- `KVM: arm64: Unconditionally save+flush host FPSIMD/SVE/SME state`：在加载 vCPU 时无条件保存和刷新主机的 FPSIMD/SVE/SME 状态。
- `KVM: arm64: Remove VHE host restore of CPACR_EL1.ZEN` 和 `KVM: arm64: Remove VHE host restore of CPACR_EL1.SMEN`：移除冗余的状态保存和恢复逻辑，简化代码。
- `KVM: arm64: Remove host FPSIMD saving for non-protected KVM`：删除不再需要的主机 FPSIMD 状态保存代码。

这些进展表明，开发者们正在积极优化 KVM 的性能和稳定性，确保在不同模式下的状态管理更加高效。
