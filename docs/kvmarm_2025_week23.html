46 messages 5 participants Started: 2025-06-05
本邮件线程讨论了一个关于 KVM 的补丁系列，主要集中在支持通过 `guest_memfd` 进行共享内存的映射，以增强软件保护虚拟机的安全性。

1. **原始补丁内容**：该补丁系列（[PATCH v11 00/18]）的核心目的是在主机上映射由 `guest_memfd` 支持的内存，以便为软件保护的虚拟机提供更好的安全性。补丁包括对 `KVM` 的多个功能进行重构和扩展，特别是针对 `arm64` 和 `x86` 架构的支持。

3. **本周新讨论和进展**：本周的讨论主要集中在补丁的具体实现和功能验证上。Fuad Tabba 提出了对补丁的更新，包括添加共享内存支持的功能、重构代码以简化逻辑，以及确保在不同虚拟机类型下的兼容性。参与者们对补丁的细节进行了审查和讨论，确认了补丁的可行性，并提出了一些优化建议，如使用 `READ_ONCE()` 来确保内存访问的正确性。此外，补丁还引入了新的 KVM 能力 `KVM_CAP_GMEM_SHARED_MEM`，以指示 `guest_memfd` 支持共享内存的能力。

[PATCH 00/17] ARM64 PMU Partitioning

34 messages 3 participants Started: 2025-06-02
本邮件线程讨论了 ARM64 PMU（性能监控单元）分区的实现及其相关补丁。以下是讨论的主要内容：

1. **原始补丁/问题内容**：
本系列补丁（共17个）旨在实现 ARM64 的分区 PMU 方案，允许将 PMU 计数器分为主机保留和客户机保留两组。通过内核命令行参数和 vCPU ioctl，可以启用此功能，从而减少客户机使用 PMU 功能时的开销。

2. **之前讨论要点**：
- 该实现利用了新 CPU 特性，允许客户机直接访问常用的 PMU 寄存器，显著降低了性能监控工具（如 `perf`）在客户机中的开销。
- 讨论中提到，分区 PMU 的实现需要在 VHE（虚拟化扩展）模式下进行，且需要 Fine Grain Traps（FGT）支持，以避免对常用寄存器的陷入。

3. **本周的新讨论、进展或结论**：
- 本周的讨论集中在补丁的具体实现上，包括如何在分区 PMU 中处理 IRQ、如何在上下文切换时保存和恢复寄存器、以及如何通过 ioctl 接口允许用户空间请求 PMU 分区。
- 参与者对补丁的设计提出了改进建议，例如如何处理 PMU 事件的调度和如何确保在分区情况下不会影响主机性能。
- 讨论中还涉及到如何避免在不同物理 CPU 上的 vCPU 之间出现不一致的 PMU 状态。

总的来说，本次讨论推动了 ARM64 PMU 分区功能的实现，强调了性能优化和系统稳定性的重要性。

[PATCH v2 0/4] KVM: arm64: Add attribute to control GICD_TYPER2.nASSGIcap

22 messages 4 participants Started: 2025-05-30
本邮件线程讨论了针对 KVM（Kernel-based Virtual Machine）在 arm64 架构下的系统寄存器访问器的改进。以下是讨论的主要内容：

1. **原始 patch/问题的内容**：最初的补丁（PATCH v2 0/4）旨在添加一个属性，以控制 GICD_TYPER2.nASSGIcap。该补丁的目的是允许用户空间在虚拟机初始化之前控制虚拟处理单元（vPE）的分配，特别是在 GICv4.1 系统上。

2. **之前的讨论要点**：历史讨论中，参与者提到需要简化用户空间与内核之间的接口，确保在初始化 VGIC（虚拟通用中断控制器）之前，用户空间可以配置相关属性。此外，讨论中还提到了一些关于如何处理 GICv4 和 GICv4.1 的细节。

3. **本周的新讨论、进展或结论**：本周的讨论主要集中在对系统寄存器访问器的重构上，包括引入特定的赋值和读-修改-写（RMW）访问器。这些改进旨在确保在对系统寄存器进行赋值时，能够正确应用 RESx 掩码，并且避免不必要的重复工作。参与者对这些改动表示支持，并讨论了如何在未来的版本中整合这些改动。最终，Marc Zyngier 确认将这些补丁应用到修复中，标志着讨论的积极进展。

总体而言，邮件线程展示了对 KVM arm64 系统寄存器访问的持续改进和优化，旨在提升性能和可靠性。

2. **之前讨论要点**：历史讨论中提到，KVM 的现有处理方式虽然能有效阻止损坏内存的传播，但缺乏优雅的恢复机制。建议通过 KVM_SET_VCPU_EVENTS API 将 SEA 重放给故障的虚拟 CPU，从而限制损坏范围，并允许虚拟机继续运行。此外，VMM（虚拟机监控器）可以跟踪 SEA 事件，并在达到一定阈值时重启虚拟机。

3. **本周新讨论及进展**：本周的讨论中，Jiaqi Yan 提出了多个补丁，主要包括：
- 引入 KVM_CAP_ARM_SEA_TO_USER 功能，允许用户空间在处理 SEA 时接收 KVM_EXIT_ARM_SEA 事件，并提供详细的故障信息。
- 增加 KVM_EXIT_ARM_SEA 作为新的退出原因，用户空间可以通过该信息来处理 SEA。
- 允许用户空间通过 KVM_SET_VCPU_EVENTS 注入外部指令中止（IABT）和数据中止（DABT）到虚拟 CPU。
- 提供了自测用例以验证新功能的正确性，包括如何处理和注入 SEA。

这些补丁和讨论旨在提升 KVM 在处理内存错误时的灵活性和稳定性，减少对客户的影响。
